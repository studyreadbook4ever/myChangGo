# Kanji Wake 시행착오 보고서

> 이 문서는 정적 일본어 DB를 사용하던 v0.1의 잠금 오버레이 문제를 기록한 역사 문서입니다. 현재 v0.4 앱 이름은 `Per-Open Quest`이며 클라우드 또는 휴대폰 내장 LiteRT-LM으로 문제를 생성합니다.

## 결론

안 됐던 주범은 단어 DB 자체라기보다, 잠금 해제 후 백그라운드 서비스에서 `Activity`를 띄우는 구조였습니다. 80개 DB일 때는 로딩이 가볍고 타이밍이 좋아서 잘 보였지만, DB가 커지면서 첫 실행과 DB 초기화가 무거워졌고 Android의 백그라운드 Activity 실행 제한, 브로드캐스트 타이밍, 서비스 생존성 문제가 같이 드러났습니다.

현재 해결 방향은 `Activity` 실행이 아니라 `SYSTEM_ALERT_WINDOW` 권한을 받은 뒤 `TYPE_APPLICATION_OVERLAY`로 퀴즈 화면을 직접 덮는 방식입니다.

## 1. 80개 DB 버전이 됐던 이유

초기 버전은 `ACTION_USER_PRESENT`를 받으면 곧바로 `QuizActivity`를 시작했습니다.

```kotlin
if (intent.action == Intent.ACTION_USER_PRESENT) {
    launchLockQuiz()
}
```

```kotlin
startActivity(intent)
```

이 구조는 단순하고 빠릅니다. 80개 DB에서는 퀴즈 화면 생성과 DB 접근이 가벼워서 잠금 해제 직후 타이밍에 잘 맞았습니다. 하지만 이 방식은 Android가 항상 허용하는 보장된 잠금 방식은 아니었습니다.

## 2. DB 확장 후 병목

큰 DB 버전에서는 `vocabulary.tsv`를 asset에서 읽어 SQLite에 넣습니다.

```kotlin
seedFromAsset(db)
```

```kotlin
appContext.assets.open(VOCABULARY_ASSET)
line.split('\t')
insertWord(db, ...)
```

또 `DB_VERSION = 2`가 되면서 기존 설치 앱은 업그레이드 시 테이블을 다시 만들고 seed를 다시 넣습니다.

```kotlin
db.execSQL("DROP TABLE IF EXISTS $TABLE_WORDS")
onCreate(db)
```

이 과정이 잠금 해제 직후 첫 퀴즈 표시 타이밍과 겹치면 화면 표시가 늦어집니다. DB가 앱을 망가뜨린 게 아니라, 백그라운드 Activity 실행이라는 약한 구조가 큰 DB 로딩 때문에 더 쉽게 깨진 것입니다.

## 3. Full-screen notification 버전의 한계

중간 버전에서는 Activity 실행 실패에 대비해 full-screen notification을 추가했습니다.

```kotlin
showUnlockQuizAlert()
startActivity(intent)
```

하지만 이 코드도 결국 `ACTION_USER_PRESENT`를 받은 뒤에만 실행됩니다. 서비스가 해당 이벤트를 못 받거나 receiver가 살아 있지 않은 순간이면 알림도, Activity도 실행되지 않습니다.

그래서 상단 foreground service 알림인 `잠금 해제 뒤 일본어 한자 단어 퀴즈를 띄웁니다.`만 남고 실제 퀴즈는 안 뜨는 상황이 생겼습니다. 그 알림은 서비스 생존 표시이지, 잠금 퀴즈가 실행됐다는 뜻이 아니었습니다.

## 4. Screen on/off polling 버전의 문제

그 다음 버전에서는 `SCREEN_OFF`, `SCREEN_ON`, `USER_PRESENT`를 같이 감지했습니다. 방향은 맞았지만 빠른 재잠금/재해제에서 막히는 코드가 있었습니다.

```kotlin
if (now - lastLaunchElapsed < LAUNCH_THROTTLE_MS) return
```

`LAUNCH_THROTTLE_MS = 5_000L` 때문에 한 번 잠금 해제하고 5초 안에 다시 잠갔다 풀면 퀴즈 표시가 스스로 차단될 수 있었습니다. soft lock에서는 매 unlock cycle마다 떠야 하므로 이 throttle은 구조적으로 맞지 않았습니다.

또 이 버전도 최종 표시는 여전히 `startActivity(intent)`였기 때문에 Android background Activity 제한 문제를 완전히 피하지 못했습니다.

## 5. 현재 되는 버전의 핵심

현재 버전은 잠금 해제 후 Activity를 앞으로 끌어올리지 않고, 오버레이 View를 직접 추가합니다.

```kotlin
WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY
```

```kotlin
windowManager.addView(root, overlayParams())
```

서비스 쪽에서는 이벤트를 넓게 받습니다.

```kotlin
addAction(Intent.ACTION_SCREEN_OFF)
addAction(Intent.ACTION_SCREEN_ON)
addAction(Intent.ACTION_USER_PRESENT)
```

그리고 권한이 있으면 바로 오버레이를 띄웁니다.

```kotlin
if (Settings.canDrawOverlays(this)) {
    val currentOverlay = overlay ?: LockQuizOverlay(this).also { overlay = it }
    currentOverlay.show()
}
```

이 방식은 Android의 background Activity launch 제한에 덜 걸립니다. 사용자가 `다른 앱 위에 표시` 권한을 허용해야 하지만, soft lock 목적에는 이 구조가 더 직접적입니다.

## 6. 참고한 Android 제약

- Android는 Android 10 이후 백그라운드 Activity 실행을 제한합니다. Android 14 이상에서는 제한이 더 강합니다.
- `TYPE_APPLICATION_OVERLAY`는 `SYSTEM_ALERT_WINDOW` 특수 권한이 필요합니다.
