# 느슨하게 연결된 웹 멀티플레이 설계 가이드

RelayPlay가 겨냥하는 형태는 흔히 말하는 실시간 동기화 게임보다
**동시에 진행하는 여러 개의 싱글플레이 + 드문 상호작용**에 가깝습니다.
테트리스 점수/생존 경쟁, 리듬게임 스코어 레이스, 타이핑 레이스처럼
각자의 다음 프레임을 계산하는 데 상대 입력이 필요하지 않은 게임입니다.

## 결론부터

이 구조에서는 브라우저끼리 P2P로 연결할 이유가 거의 없습니다.

```text
플레이어 A ─ 진행도/공격 의도 → 방 서버 ─ 확정 이벤트 → 플레이어 B
플레이어 B ─ 진행도/공격 의도 → 방 서버 ─ 확정 이벤트 → 플레이어 A
```

클라이언트는 입력·렌더링·오디오·자기 게임 시뮬레이션을 로컬에서 즉시
처리합니다. 방 서버는 프레임을 돌리지 않고 다음만 담당합니다.

- 참가·인증·준비·동시 시작
- 기본 1초 간격의 진행도 중계
- 공격/방해 의도의 유효성·쿨다운·대상 검사
- 중요한 이벤트의 단일 순서 부여와 저장
- 끊겼다가 돌아온 클라이언트의 이벤트 재생
- 결과 및 리플레이 검증 확장점

이렇게 하면 서버 비용과 트래픽은 작으면서도 P2P보다 인증, 치팅 대응,
재접속, 관전자, 신고/차단 정책을 훨씬 일관되게 만들 수 있습니다.

## 어떤 데이터를 보내나

데이터를 세 종류로 나누면 설계가 단순해집니다.

| 종류 | 예시 | 처리 규칙 |
| --- | --- | --- |
| 로컬 상태 | 보드, 노트, 애니메이션, 입력 | 네트워크를 기다리지 않음 |
| 진행 스냅샷 | 점수, 콤보, 높이, 진행률 | 최신값만 중요, 누락 허용 |
| 확정 이벤트 | 시작, 승인된 방해, 종료 | 저장 후 중계, 순서대로 재생 |

진행도는 기본 1,000ms마다 보내고 화면에서는 두 값 사이를 보간하면
충분한 경우가 많습니다. 20ms마다 점수를 보낸다고 권위가 생기는 것은
아니며 트래픽과 공격 면적만 커집니다.

## 상호작용은 도착 즉시 적용하지 않는다

네트워크 지연은 매번 다릅니다. 따라서 상대의 공격이 패킷 도착 순간에
발동하면 같은 상황에서도 결과가 달라지고 모바일망 사용자가 손해를 봅니다.

서버가 공격을 승인할 때 적용 시점을 미래로 정합니다.

- 테트리스 계열: 현재 블록이 고정된 뒤, 다음 블록 생성 전
- 고정 틱 게임: 현재보다 충분히 먼 특정 틱
- 리듬게임: 미래의 박자 또는 마디
- 일반 레이스: 750ms 뒤 경고 종료 시점

너무 늦게 도착했을 때도 즉시 발동시키지 말고 `폐기`, `다음 경계로 이월`
같은 정책을 설정으로 명시해야 합니다.

## 테트리스 계열

동일한 규칙 버전, 시드, 고정 틱, 입력 로그를 사용하면 사후 검증이
상대적으로 쉽습니다. 상대에게는 점수·라인·보드 높이 정도를 진행도로
보내고, 공격은 서버가 승인한 뒤 `다음 블록 고정` 같은 논리적 경계에
적용합니다. 랭크전이라면 입력 리플레이로 라인 클리어와 공격 발생 조건을
다시 계산할 수 있습니다.

## 리듬게임 계열

서버 시간을 여러 번 ping/pong하여 오프셋과 불확실성을 추정하고, 시작은
몇 초 뒤의 서버 시각으로 예약합니다. 실제 곡 진행은 사용자 제스처로
오디오가 열린 뒤 `AudioContext.currentTime`에 고정하고, 화면 렌더링은
`requestAnimationFrame`으로 따로 처리합니다. 60Hz와 120Hz에서 점수가
달라지면 안 됩니다.

일반 브라우저는 물리적인 키 입력 시각을 위조할 수 있으므로 “웹만으로
완전한 치팅 방지”라고 주장하면 안 됩니다. 차트·점수 계산의 일관성 검증,
이상치 탐지, 리플레이 검사는 가능하지만 신뢰 수준을 구분해야 합니다.

## 모바일/PC 선택

권장 기본값은 **동일한 시뮬레이션·점수·프로토콜 + 다른 입력·레이아웃**입니다.

### 유니버설

대부분의 캐주얼 게임에 적합합니다. 터치/키보드/포인터 능력을 실제로
감지하고 반응형 레이아웃, safe-area, 큰 터치 목표, 오디오 잠금 해제,
저사양 효과 감소를 적용합니다. 기기 이름보다 입력 능력을 기준으로 봅니다.

### 모바일 우선

세로 화면과 한두 엄지 조작이 핵심일 때 선택합니다. 동적 viewport,
터치 취소, 백그라운드 전환, 스크롤 방지 영역을 먼저 설계합니다. PC판은
같은 규칙을 쓰되 모바일 UI에 키보드를 억지로 끼워 넣지 않습니다.

### PC 우선

고밀도 채보, 정밀 키보드 입력, 큰 관전 화면처럼 터치로 공정하게 옮기기
어려운 경우에 선택합니다. 모바일 지원을 숫자상 체크하기보다 해당 입력
모델이 실제로 가능한지 명확히 공개하는 편이 낫습니다.

캐주얼 매칭은 크로스플레이를 열고, 랭크전은 실제 입력 종류별 성과 데이터를
수집한 뒤 차이가 확인될 때만 `통합`, `입력 종류별`, `플랫폼별` 풀로
나누는 것이 좋습니다. 입력 차이를 보정하려고 플랫폼별 물리나 점수 공식을
갈라 버리면 리플레이와 밸런스 관리가 더 어려워집니다.

## 서버리스 배치

한 방을 하나의 Cloudflare Durable Object가 담당하게 하면 메시지가 자연스럽게
한 줄로 정렬됩니다. 이 객체는 60Hz 루프를 돌지 않습니다. 메시지가 왔을 때
검사·저장·중계하고, 유휴 상태에서는 hibernatable WebSocket으로 잠들 수
있습니다. 대규모 공유 월드가 아니라 이 게임 형태에 잘 맞는 이유입니다.

## 출시 순서

1. 동시 시작 + 진행도 비교만 먼저 완성합니다.
2. 경고가 보이는 단 하나의 방해 이벤트를 미래 경계에 추가합니다.
3. 끊김/복귀와 이벤트 재생을 검증합니다.
4. 확정 이벤트 로그와 결과 해시를 남깁니다.
5. 보상이 걸린 랭크 전에 리플레이 검증기를 붙입니다.
6. 실제 지표를 보고 진행 전송 주기와 입력별 매칭을 조정합니다.

설정과 구현 방법은 [설정 문서](../configuration.md),
[프로토콜 문서](../protocol.md), [보안 문서](../security.md)를 참고하세요.
